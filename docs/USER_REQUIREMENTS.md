# 📚 다독 리뷰 (Dadoc Review) 사용자 요구사항 정의서

본 문서는 개발자(사용자)가 AI 어시스턴트에게 전달한 핵심 요구사항과 학습 및 코딩 철학을 정리한 문서입니다.

---

## 1. 프로젝트 비전 및 목표
- **서비스 명:** 다독 리뷰 (Dadoc Review)
- **핵심 가치:** 흩어진 도서 리뷰를 AI로 요약하여 통찰을 제공하고, 실제 대출 가능한 도서관 정보를 연결함으로써 "책 선택부터 대출까지"의 과정을 최적화함.

---

## 2. 핵심 성공 기준 (Success Criteria) 🌟
사용자님께서 가장 강조하신 **실제 구현의 증거**들입니다.
1.  **DB 데이터 영속성:** 실제 도서(Book) 데이터를 데이터베이스에 성공적으로 저장하고 관리할 수 있어야 함.
2.  **시각적 확인:** 저장된 데이터를 API 응답이나 화면(UI)을 통해 눈으로 직접 확인할 수 있어야 함.
3.  **단계별 검증:** 이론에 머물지 않고 각 기능을 구현할 때마다 실제로 작동하는지 테스트하며 나아감.

---

## 3. 학습 및 코딩 철학 (Working Principles) 💡
단순히 코드를 완성하는 것이 아니라, 다음의 원칙을 지키며 공부합니다.

### 📁 체계적인 파일 분리 (Architecture)
- `database.py`, `models.py`, `schemas.py`, `crud.py`, `main.py` 등으로 역할을 분리하여 현업 수준의 깔끔한 구조를 지향함.

### ✍️ [파일 역할] 기록 (Documentation)
- 모든 주요 파일 상단에 해당 파일의 목적, 역할, 포함된 기술 개념을 한국어 주석으로 상세히 기록함.
- 코드를 복사하는 것이 아니라 '원리'를 이해하며 작성함.

### 🚀 프로토타입 중심 개발 (Hands-on)
- 완벽한 이론보다는 "작동하는 코드"를 먼저 만들고, 이를 개선해 나가는 반복적(Iterative) 개발 방식을 취함.

---

## 4. 기능적 요구사항 요약
- **데이터 수집:** 외부 도서 API 및 온라인 서점 크롤링.
- **AI 분석:** 리뷰 데이터를 Gemini API 등으로 요약(장단점, 핵심 키워드).
- **도서관 연동:** 위치 기반 공공도서관 대출 현황 조회.
- **검색 시스템:** ISBN 등을 통한 통합 도서 검색.

## 6. AI 어시스턴트의 역할 가이드라인 🤖
- **코딩의 주체:** 모든 실제 코드는 **사용자(USER)가 직접** 작성한다.
- **조력의 한계:** AI는 코드를 직접 파일에 기록하지 않고(명시적 요청 제외), 구현을 위한 **가이드, 로직 설명, 힌트 제공**에 집중한다.
- **상세 가이드 방식:** 힌트는 한 번에 많이 주지 않고, 사용자님이 스스로 해결할 수 있는 **작은 단계(Small Steps)**로 나누어 제공한다.
- **공식 문서 참조:** 가이드를 제공할 때는 반드시 해당 기술(FastAPI, Pydantic, SQLAlchemy 등)의 **공식 문서 링크**를 첨부하여 근거를 명시한다.
- **커밋 알림:** 작업이 의미 있는 단위로 완료될 때마다 사용자에게 **Git 커밋을 제안**하여 좋은 개발 습관을 기르도록 돕는다.
- **핵심 목표:** 사용자가 코드를 직접 입력하며 기술적 원리(FastAPI, SQLAlchemy 등)를 체득하는 과정을 보조한다.
